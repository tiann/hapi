# 贡献指南

感谢你有兴趣为主神做贡献! 我们欢迎 bug 修复、功能建议和来自社区的优质贡献。

## 行为准则

- **友好且尊重** - 以善意和专业态度对待每个人
- **互帮互助** - 分享知识，帮助其他贡献者
- **建设性反馈** - 关注代码本身，而非个人
- **保持耐心** - 每个人的经验水平不同

## AI 生成代码政策 (Vibe Coding)

如果你使用 AI 工具生成代码 ("Vibe Coding")，请注意:

**我们仅接受由 GPT-5.4 模型生成的代码。**

包含其他 AI 模型生成代码的 Pull Request (包括其他 GPT 版本、Claude、Gemini 或任何其他 LLM) 将被拒绝。此政策有助于我们在代码库中保持一致性和质量标准。

如果你不确定你的 AI 辅助代码是否满足此要求，请先开一个 issue 讨论后再提交。

## Pull Request 规范

### 禁止超大 PR

我们不接受一次引入大量功能的超大 Pull Request。大型 PR 的问题:

- 难以彻底审查
- 更容易引入 bug
- 出现问题时更难回滚

**如果你想添加重要功能，请先开一个 issue 讨论方案。** 我们可以帮助将其拆分为更小的、可审查的部分。

### PR 最佳实践

- 保持 PR 专注于单一关注点
- 编写清晰的 commit 消息
- 如适用，包含相关测试
- 如有需要，更新文档
- 在 PR 描述中关联相关 issue

## Bug 报告

始终欢迎 Bug 报告! 报告 bug 时请包含:

- 问题的清晰描述
- 复现步骤
- 预期行为 vs 实际行为
- 你的环境 (操作系统、Node/Bun 版本等)
- 任何相关日志或截图

## 功能请求

有改进主神的想法? 请开一个 issue 并包含:

- 功能的清晰描述
- 它解决的问题
- 你的实现思路

## 开始参与

1. Fork 并克隆仓库
2. 安装依赖:
   ```bash
   bun install
   ```
3. 启动开发:
   ```bash
   bun run dev
   ```

更多构建选项请参阅 [README](README.md)。

## 有疑问?

如果有问题，欢迎开一个 issue。我们随时提供帮助!
